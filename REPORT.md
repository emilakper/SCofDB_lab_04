## 1. Постановка сценария
_TODO: Опишите сценарий \"запрос на оплату -> обрыв сети -> повторный запрос\"._

В лабораторной работе моделируется классическая проблема распределённых систем - сценарий "запрос на оплату -> обрыв сети -> повторный запрос":

1) Клиент отправляет POST-запрос на оплату заказа.

2) Сервер начинает обработку, но до получения клиентом ответа происходит обрыв сети (таймаут, потеря соединения и т.д.).

3) Клиент, не получив подтверждения, повторяет тот же запрос.

Без защиты:

- Сервер не знает, что это повторный запрос.
- Оба запроса обрабатываются независимо, что приводит к двойному списанию средств с одного заказа.

Почему возможна повторная обработка:

- Отсутствие механизма идемпотентности на уровне API.
- Сервер не может отличить новый запрос от повторного без специального идентификатора.
- Сетевая инфраструктура не гарантирует доставку ответа.

## 2. Реализация таблицы idempotency_keys
_TODO: Приведите итоговый DDL таблицы и краткое объяснение колонок._
Итоговый DDL таблицы:

```sql
CREATE TABLE idempotency_keys (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    idempotency_key VARCHAR(255) NOT NULL,
    request_method VARCHAR(16) NOT NULL,
    request_path TEXT NOT NULL,
    request_hash TEXT NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'processing',
    status_code INTEGER,
    response_body JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMP NOT NULL DEFAULT (NOW() + INTERVAL '7 days'),
    
    CONSTRAINT idempotency_status_check CHECK (status IN ('processing', 'completed', 'failed')),
    CONSTRAINT uniq_idempotency_key_method_path UNIQUE (idempotency_key, request_method, request_path)
);
```

**Описание колонок:**
- `idempotency_key`: сам ключ идемпотентности из заголовка запроса.
- `request_method` и `request_path`: идентифицируют конкретный endpoint (POST /api/payments/retry-demo).
- `request_hash`: SHA256 хэш тела запроса для обнаружения повторного использования ключа с другим payload.
- `status`: состояние обработки (`processing` — в обработке, `completed` — завершено, `failed` — ошибка).
- `status_code` и `response_body`: кэш ответа для повторных запросов.
- `expires_at`: время жизни записи (7 дней) для автоматической очистки.

**Ограничения и индексы:**
- `uniq_idempotency_key_method_path`: уникальность ключа в рамках endpoint — защита от дубликатов на вставке.
- `idx_idempotency_lookup`: составной индекс для быстрого поиска по ключу + методу + пути.
- `idx_idempotency_expires`: индекс для эффективной очистки просроченных записей.
- Триггер `trig_idempotency_updated_at`: автоматическое обновление `updated_at` при изменении записи.

## 3. Реализация middleware

**Алгоритм работы `IdempotencyMiddleware` по шагам:**

1. **Чтение `Idempotency-Key`:**
   - Проверяется, что метод запроса POST и путь начинается с `/api/payments/`.
   - Из заголовков извлекается `Idempotency-Key`. Если ключ отсутствует — запрос передаётся дальше без обработки.

2. **Проверка существующей записи:**
   - Вычисляется хэш тела запроса через `build_request_hash()`.
   - В транзакции с `FOR UPDATE` выполняется поиск записи по ключу, методу и пути.
   - Блокировка строки предотвращает race condition при параллельных запросах с одинаковым ключом.

3. **Обработка кейса "тот же key + другой payload":**
   - Если запись найдена, но `request_hash` не совпадает с текущим — возвращается `409 Conflict`.
   - Это защищает от случайного или злонамеренного переиспользования ключа.

4. **Обработка статусов:**
   - `completed`: возвращается кэшированный ответ из `response_body` с заголовком `X-Idempotency-Replayed: true`.
   - `processing`: возвращается `409 Conflict` (запрос уже обрабатывается).
   - Запись не найдена: создаётся новая запись со статусом `processing`.

5. **Сохранение результата первого запроса:**
   - После выполнения бизнес-логики перехватывается ответ.
   - Запись в БД обновляется: статус `completed`, сохраняются `status_code` и `response_body`.

6. **Возврат кэшированного ответа при повторе:**
   - При повторном запросе с тем же ключом срабатывает шаг 4 для статуса `completed`.
   - Клиент получает тот же ответ, что и при первом запросе, без повторного выполнения логики оплаты.

## 7. Сравнение с решением из ЛР2 (FOR UPDATE)

**Сравнение подходов:**

| Характеристика | FOR UPDATE (ЛР2) | Idempotency-Key (ЛР4) |
|----------------|------------------|------------------------|
| **Цель механизма** | Защита от race condition (конкурентные запросы) | Защита от повторных запросов из-за сетевых проблем |
| **Что происходит при повторе запроса** | Второй запрос видит, что заказ уже оплачен → бизнес-ошибка (success=False) | Второй запрос получает кэшированный ответ первого запроса (success=True) |
| **HTTP статус при повторе** | 200 (но success=False в теле) | 200 + заголовок X-Idempotency-Replayed |
| **Где обеспечивается гарантия** | Уровень БД (блокировка строк) | Уровень API-слоя (кэширование ответов) |
| **Что защищает** | Одновременные запросы к одному ресурсу | Повторные отправки одного запроса клиентом |
| **Пример сценария** | Два параллельных запроса от разных клиентов | Один клиент повторяет запрос из-за таймаута |

**Нужно ли использовать оба механизма вместе:**
Да. Эти подходы решают разные задачи и отлично дополняют друг друга:
- `FOR UPDATE` защищает от ситуации, когда два запроса (возможно, с разными ключами идемпотентности) пытаются одновременно оплатить один заказ.
- `Idempotency-Key` защищает от повторных отправок одного и того же запроса одним клиентом.

В production-системах оба механизма должны использоваться совместно для обеспечения максимальной надёжности платежей.


## 8. Выводы

1. **Проблема повторных запросов реальна:** Без защиты простой сетевой сбой может привести к двойной оплате, что критично для платёжных систем.

2. **Idempotency-Key — стандарт де-факто:** Использование заголовка `Idempotency-Key` с кэшированием ответов — эффективный и понятный способ защиты от retry-сценариев, принятый в индустрии (Stripe, PayPal и др.).

3. **Важность проверки payload:** Хранение хэша тела запроса критически важно — без этой проверки злоумышленник или ошибка в коде могли бы переиспользовать ключ с другими параметрами.

4. **Разные инструменты для разных задач:** `FOR UPDATE` и идемпотентность не конкурируют, а дополняют друг друга. Первый защищает от конкурентного доступа, второй — от повторных отправок.

5. **Прозрачность для клиента:** Заголовок `X-Idempotency-Replayed` позволяет клиенту понять, что ответ пришёл из кэша, что упрощает отладку и мониторинг.

6. **Очистка данных:** Обязательно нужно предусматривать механизм удаления старых записей (`expires_at` + фоновая задача или триггер), чтобы таблица `idempotency_keys` не росла бесконечно.
